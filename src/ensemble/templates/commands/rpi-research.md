# RPI Research - 要件解析・技術調査フェーズ

## 目的
要件解析・技術調査・実現可能性評価を行い、実装前のリスクを洗い出す。

## 使用方法
```
/rpi-research <Issue URL or タスク説明>
```

## 処理フロー

1. **要件の明確化**
   - 提供された情報から要件を抽出
   - 曖昧な点・不明瞭な点をリストアップ
   - 追加で確認が必要な事項を特定

2. **技術的実現可能性の評価**
   - 現在の技術スタックで実現可能か
   - 新規ライブラリ・ツールの導入が必要か
   - 技術的な制約・課題はあるか

3. **依存関係の特定**
   - 既存コードへの影響範囲
   - 他機能との依存関係
   - 外部サービス・APIへの依存

4. **リスク分析**
   - 技術的リスク（実装難易度、パフォーマンス、スケーラビリティ）
   - スコープリスク（要件の曖昧さ、変更の可能性）
   - 運用リスク（保守性、テスト可能性）

5. **GO/NO-GO判定の提示**
   - GO推奨: 実現可能性が高く、リスクが管理可能
   - HOLD: 追加調査が必要、または依存関係の解消待ち
   - NO-GO: 実現不可能、または代替案の検討が必要

## 出力先
`rpi/research-{タスク名}.md` に調査結果を保存

## 出力フォーマット例

```markdown
# Research: {タスク名}

## 要件の明確化
- 明確な要件:
  - ...
- 曖昧な点・要確認事項:
  - ...

## 技術的実現可能性
- 実現可能性: [高/中/低]
- 必要な技術スタック:
  - ...
- 技術的課題:
  - ...

## 依存関係
- 影響を受ける既存機能:
  - ...
- 依存する外部サービス:
  - ...

## リスク分析
### 技術的リスク
- ...

### スコープリスク
- ...

### 運用リスク
- ...

## 判定
**[GO / HOLD / NO-GO]**

理由: ...

## 次のステップ
判定がGOの場合: `/rpi-plan` で計画フェーズに進んでください
判定がHOLDの場合: 以下の追加調査を実施してください
判定がNO-GOの場合: 代替案を検討してください
```

## 注意事項
- 調査結果は必ずファイルに保存する（セッション終了後も参照可能にするため）
- 曖昧語を使わず、具体的な数値・ファイル名・行数で記述する
- リスクは必ず定量化する（影響範囲、工数、確率など）
